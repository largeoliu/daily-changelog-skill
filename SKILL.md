# Daily Changelog

这个 skill 是 vendor-neutral 的，目标是让任何能读取本地文件并执行仓库脚本的 agent，都能把代码变更整理成中文产品更新日志。

## 何时使用

当用户提到以下需求时使用：

- 更新日志
- changelog
- release notes
- 产品变更说明
- 今天做了什么

## 调用方视角

- 调用方只需要表达：时间范围、项目路径、可选输出文件路径
- 调用方不需要执行 Python 命令
- 调用方不需要感知 `themes.md`、`draft.md` 等任何中间产物
- 对调用方来说，这个 skill 应当是黑盒：正式执行路径只允许公开入口 `scripts/daily_changelog_run.py`
- 调用方 agent 成功时直接得到最终 changelog，失败时只返回失败原因且不保存最终文件
- 调用方 agent 不得编辑 `draft.md`、`themes.md`、最终输出文件，也不得基于报错清单手工删改产物
- 调用方 agent 不得为了修补产物而临时创建脚本、批量删行、正则替换生成结果
- 调用方 agent 不得默认执行单日试探、分步调试、中间产物检查或 `--keep-artifacts` 一类调试动作
- 调用方 agent 不得读取或依赖内部实现脚本，不得自行拆仓、拆年份、拆月份或改写运行策略

可直接触发的自然语言示例：

- 使用 `$daily-changelog`，基于 `/path/to/project` 在 `2026-03-01` 到 `2026-03-07` 的代码变更，生成中文产品更新日志，并保存到 `/path/to/product-changelog.md`
- 使用 `$daily-changelog`，基于 `/path/to/project` 今天的代码变更生成中文产品更新日志

## 输入与默认值

- 日期支持单日或日期区间；未提供日期时默认今天
- “上线至今” / “自上线以来” 直接映射为 `earliest`
- 单个路径优先按 `--repo-path` 处理，可为单仓库或聚合目录
- 用户明确给出多个独立仓库路径时，再改用 `--repos`
- 当日期跨度超过 31 天时，skill 会在内部自动分段执行并统一汇总；调用方不要自行按年、按月或按单日拆分
- 输出顺序只定义最终多个日期块的排列方式：`desc` 为最新在前，`asc` 为最早在前；未指定时默认使用 `desc`
- 单日和多日使用同一套逐日流程；单日只是只处理一个日期

## 工作流

1. 先解析日期列表、仓库路径和输出顺序。
2. 对单路径或聚合目录，优先让公开入口使用 `--repo-path` 自动识别仓库；只有用户明确给了多个独立路径时才使用 `--repos`。
3. 当日期范围较大时，skill 应在内部自动按窗口分段执行并统一归并，调用方 agent 不得自行按年份或月份拆分任务。
4. 正式生成只使用公开黑盒入口。`themes.md`、`draft.md` 以及所有内部脚本都属于内部实现细节，不应作为调用方决策依据。
5. 不要用 `head`、`tail`、截断读取等方式裁剪分析输出；结论必须基于完整分析结果、关键证据和文件上下文，而不是 commit message。
6. 每天若返回 `NO_CHANGES`，则跳过该日期；若返回 `REPO_DISCOVERY_ERROR`，则向用户说明路径范围不可解析并停止。
7. 读取 `references/writing_rules.md`，把技术证据翻译成产品语言；如果报告里出现主题归并提示，优先按主题合并。
8. 读取 `references/changelog_template.md`，先生成单日日期块。只有当天确有实质产品变更时才输出该日期。
9. 最终结果必须同时通过结构校验和语义校验；任一失败都不得写最终文件。
10. 遇到校验失败或环境限制时，应直接返回失败原因，不得通过手改 `draft.md`、补写脚本或跳过校验来冒充成功结果。
11. 除非用户明确要求调试 skill 本身，否则不要改变日期范围做单日试探，也不要读取中间产物、内部脚本或 repo 子范围来决定是否继续运行。

## 失败后的正确动作

- 优先返回 `REPO_DISCOVERY_ERROR`、校验失败摘要、路径不存在或路径不可访问等明确原因
- 如果目标仓库路径或输出路径不在当前 agent workspace 可访问范围内，应先报告环境限制，不进入生成流程
- 如果语义校验失败，应回到主题归并规则、生成规则或输入路径排查，而不是修补 markdown 成品

## 参考文件

- `references/writing_rules.md`
  用于查看产品化改写规则、禁写项、主题归并原则和常见技术信号的翻译方式。
- `references/changelog_template.md`
  只在撰写最终 markdown 时读取；它定义了允许的标题结构和最小模板。

## 输出硬约束

1. 最终文件第一条非空内容必须是 `# 产品更新日志`
2. 日期标题必须严格为 `## YYYY-MM-DD`
3. 只能使用这 4 个分类：`✨ 新功能`、`🔄 功能变更`、`🔧 技术改造`、`🐛 Bug 修复`
4. 空日期不输出，空分类不输出
5. 每条内容都必须说明“用户能做什么”或“系统发生了什么变化”，并带一个场景锚点
6. 不得输出文件路径、文件名、类名、组件名、接口路径、DTO、VO、SQL、Controller、Service、Mapper、Req、Resp
7. 不得输出 commit message 风格前缀、统计信息、生成时间、工具名、仓库列表
8. 同一主题下的前后端、SQL、DTO、页面支撑文件默认应归并成一条产品描述，不要按技术文件拆条

## 校验与失败处理

- `scripts/daily_changelog_run.py`
  唯一公开入口。它会在内部完成完整生成流程，并且只有全部通过时才写最终文件。
- 若产品含义无法确认，宁可降级成抽象业务描述，也不要输出技术细节。
- 如果校验失败，不要保存无效结果；先重写、再降级，最后才删除对应条目。
