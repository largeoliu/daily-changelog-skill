# 产品化改写规则

分析报告是证据，不是最终文案。最终输出必须写成业务方能直接阅读的产品更新日志。

## 必须遵守

1. 每条内容必须说明“用户能做什么”或“系统发生了什么变化”
2. 每条内容必须带一个场景锚点，例如页面、模块、角色、流程或操作场景
3. 不要照抄 commit message、diff、类名、路径或分析报告原文
4. 如果无法确认产品含义，优先降级成抽象业务描述，不要输出技术细节
5. 同一主题下的前后端、数据库、DTO、支撑组件默认要归并，不要按文件逐条罗列

## 禁写项

- 文件路径、文件名、类名、方法名、组件名
- 接口路径、英文路由片段、分支名、任务号
- DTO、VO、Req、Resp、SQL、Controller、Service、Mapper
- `feat(`、`fix(`、`refactor(`、`chore(`、`test(` 这类提交前缀
- 统计信息、生成时间、工具名、仓库列表

## 主题归并原则

- 只有能映射到独立页面、独立入口、独立菜单或独立运营流程的主题，才适合单列
- 纯后端入口、DTO/VO/Query、Repository、Mapper、前端子组件、图表、筛选器、导出列、状态层、数据层，默认只作为支撑证据
- 如果同一主题下还有多个用户可感知的小能力，不要丢失；把它们并入一条主功能里，用并列短语或“并支持……”表达

## 技术信号到产品含义

| 技术信号 | 常见产品含义 |
|---|---|
| 新增前端路由 | 新页面或新入口上线 |
| 新增 HTTP 接口 | 可能新增某个操作入口或流程能力，需结合页面和场景确认 |
| 新增 Dubbo 接口 | 多为内部支撑能力，默认不单列 |
| 新增定时任务 | 多为后台自动化处理，只有用户可感知时才写 |
| SQL 迁移 | 数据结构、统计口径或底层支撑能力变化 |
| Query 新增字段 | 新筛选条件或统计维度变化 |
| DTO / VO 新增字段 | 展示字段、返回内容或请求参数变化 |
| 前端按钮、标题、表格列 | 页面操作入口、字段展示或筛选项变化 |

## 改写示例

- 错误：新增了 `/api/example/action` 接口
- 正确：详情页新增操作入口，用户可直接发起处理流程

- 错误：前端新增 ExamplePage 组件
- 正确：新增申请页面，用户可填写信息并提交

- 错误：DTO 字段调整，涉及 OrderController
- 正确：详情页展示字段和返回信息已同步调整

- 错误：SQL 优化
- 正确：相关查询效率提升，列表加载更稳定

## 条目质量检查

生成每条日志前，至少自检这几个问题：

- 用户读完这句，能不能知道变化发生在哪个场景？
- 用户读完这句，能不能知道多了什么能力、少了什么问题，或系统哪里更稳定？
- 这句里有没有任何技术名、路径、代码对象名？
- 这条是不是把同一主题拆得过碎了？
- 这条是否只是 DTO、异常处理、共享字段、参数解析等支撑证据，却被误写成独立功能？
- 这条如果跨多天都有证据，是否已经归到“用户真正可用的最晚日期”？
